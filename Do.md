Please do not just create a QA report. Instead, make a complete plan, implement the fixes, test the app, and keep repeating that cycle until the app is fully complete and production-ready. Start by reviewing the current app, understanding the intended workflow, and identifying what is broken, missing, confusing, duplicated, or unnecessary. Then create a clear implementation plan with prioritized fixes, apply the fixes directly in the code, remove unused or redundant files, clean up unnecessary logic, and optimize the project structure.

After implementing the first round of fixes, test the app from multiple user perspectives, including a first-time user, admin user, mobile user, PDF-view-only user, direct-PDF-editing user, redacted-download user, and a user entering incomplete or incorrect information. Do not stop after one test pass. If any issue is found, fix it, then test again. Continue this plan → implement → test → fix → retest loop until all critical workflows are working correctly.

The final app must correctly support the OSHA 300A workflow from start to finish: required data entry, validation, accurate PDF field mapping, proper year formatting, correct case counts, correct number-of-days totals, correct injury and illness type totals, establishment information, employment information, embedded PDF preview, direct fillable PDF editing only when explicitly selected, save behavior, normal download, redacted download, and view-only mode. “View Only” must never trigger a download. The embedded PDF viewer must show the completed PDF inside the website, not force a new tab or automatic download. The normal and redacted downloaded PDFs must be flattened/read-only and not fillable. Only the “Edit Directly on PDF” option should open a fillable version.

Also verify the UI and code quality. Make sure the layout is clean, responsive, accessible, and easy to understand. Buttons must be clearly labeled and perform only their intended action. Remove unused code, duplicate components, dead routes, unused styles, unnecessary packages, and confusing files. Refactor repeated logic into reusable helpers or components. Make the final codebase smaller, cleaner, faster, and easier to maintain.

When finished, provide only a concise completion summary with what was fixed, what was removed, what was tested, and whether the app is ready. Do not create a long report unless there are unresolved blockers. If there are blockers, clearly state what they are and what is still required.
